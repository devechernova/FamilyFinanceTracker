# Architektur der Anwendung

Die Anwendung wurde unter Verwendung der objektorientierten Programmierung (OOP) entwickelt, um eine klare Struktur, gute Wartbarkeit und Erweiterbarkeit zu gewährleisten.

Die Architektur basiert auf einer Trennung von Datenmodellen und Geschäftslogik. Dadurch wird sichergestellt, dass einzelne Komponenten unabhängig voneinander entwickelt und erweitert werden können.

---

## 5.1 Überblick über die Architektur

Die Anwendung besteht aus mehreren zentralen Komponenten:

- Datenklassen (Modelle) zur Speicherung der Informationen  
- einer zentralen Logikklasse (**FinanceManager**)  
- einer Benutzeroberfläche in Form eines Konsolenmenüs  

---

## 5.2 Verwendete Klassen

### User

Die Klasse **User** repräsentiert einen Benutzer der Anwendung.

**Attribute:**
- Id  
- Name  
- Role (Parent / Child)  

Die Rolle bestimmt die Zugriffsrechte innerhalb der Anwendung.

---

### Transaction

Die Klasse **Transaction** speichert Informationen über finanzielle Transaktionen.

**Attribute:**
- Id  
- Amount  
- Type (Einnahme / Ausgabe)  
- CategoryId  
- UserId  
- Date  

Jede Transaktion ist einem Benutzer und einer Kategorie zugeordnet.

---

### Category

Die Klasse **Category** dient zur Gruppierung von Transaktionen.

**Attribute:**
- Id  
- Name  

Sie ermöglicht eine strukturierte Analyse der Ausgaben.

---

### Budget

Die Klasse **Budget** verwaltet Budgetlimits für Kategorien.

**Attribute:**
- CategoryId  
- Limit  

Sie wird verwendet, um Ausgaben zu überwachen und Überschreitungen zu erkennen.

---

### FinanceManager

Der **FinanceManager** bildet die zentrale Logik der Anwendung.

**Aufgaben:**
- Verwaltung aller Benutzer  
- Verwaltung aller Transaktionen  
- Berechnung des Kontostands  
- Filterung der Daten  
- Überprüfung von Budgetlimits  

**Wichtige Methoden:**
- AddTransaction()  
- GetBalance()  
- FilterTransactions()  
- GetByCategory()  
- GetByUser()  
- CheckBudgetLimit()

## 5.4 Rollen- und Zugriffskonzept

Ein wichtiger Bestandteil der Architektur ist die Implementierung von Benutzerrollen:

- **Parent:** Zugriff auf alle Funktionen und Daten  
- **Child:** Zugriff nur auf eigene Transaktionen  

Diese Trennung sorgt für eine klare Zugriffskontrolle und erhöht die Realitätsnähe der Anwendung.

---

## 5.5 Vorteile der gewählten Architektur

Die gewählte Architektur bietet mehrere Vorteile:

- klare Strukturierung des Codes  
- einfache Erweiterbarkeit (z. B. Umwandlung in eine Webanwendung)  
- gute Wartbarkeit  
- Trennung von Daten und Geschäftslogik
